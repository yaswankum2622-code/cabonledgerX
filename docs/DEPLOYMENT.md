# Deployment

## Current Deployment Posture

The repository is designed for local demo deployment, not full production hosting. The source of truth is parquet output generated by the build scripts.

This means the current operational model is:

- build local data artifacts
- run the Streamlit dashboard
- run the FastAPI service

That is sufficient for:

- portfolio demos
- recruiter reviews
- technical walkthroughs
- local product screenshots

## Local Setup

```powershell
py -3.11 -m venv .venv
.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install -e .
```

## Required Build Artifacts

Before serving the app, build the processed outputs:

```powershell
python scripts/build_processed_tables.py
python scripts/build_emissions_baseline.py
python scripts/build_forecast_and_assessment.py
python scripts/build_risk_and_contradictions.py
python scripts/build_intervention_scenarios.py
python scripts/build_activity_and_calculated_emissions.py
python scripts/build_historical_reconstruction.py
python scripts/build_statistical_forecast_and_evaluation.py
python scripts/build_probabilistic_scoring.py
python scripts/build_scoring_reconciliation.py
python scripts/build_evidence_packs.py
```

## Running The Dashboard

```powershell
python -m streamlit run src/carbonledgerx/dashboard/app.py
```

Default URL:

- `http://localhost:8501`

## Running The API

```powershell
python -m uvicorn src.carbonledgerx.api.main:app --reload
```

Default URLs:

- `http://127.0.0.1:8000/docs`
- `http://127.0.0.1:8000/redoc`

## Public GitHub Repo Guidance

For a clean public repo:

- keep source code, scripts, tests, and docs
- keep only a small curated set of screenshots if needed
- do not commit large raw workbooks
- do not commit generated parquet outputs
- do not commit ephemeral logs, caches, or local screenshots

Recommended publishable repo contents:

- `src/`
- `scripts/`
- `tests/`
- `docs/`
- root configuration files

## If This Moves To Real Hosting

The next deployment step should not be “deploy everything as-is.” It should be:

1. separate build-time artifact generation from serve-time access
2. move processed outputs into a managed store
3. replace direct parquet reads in serving code with a stable data access service
4. add auth and request controls to the API
5. add release/version discipline for both data and models

## Non-Goals Of The Current Deployment Doc

This document intentionally does not claim:

- cloud-scale readiness
- multi-user production hardening
- database-backed serving
- enterprise security controls

Those are future work items, not part of the current public-repo promise.
